Technical Framework Specification: AutoResearch & Autonomous Heuristic Discovery
1. Framework Overview and Core Directive
The  AutoResearch Framework  is an autonomous system architected for heuristic discovery, validation, and iteration within the Spokeo identity graph. The primary objective is to engineer a transition from manual, siloed search products—where identifiers are isolated in legacy buckets—to an integrated, agent-driven discovery loop. This loop leverages LLM orchestration to synthesize disparate data signals into a unified entity view.Fundamental Architectural Goals:
* Token Efficiency:  System MUST implement  Skill-Based Routing  to ensure models consume only the specific tool schemas required for the immediate task pass.
* Retrieval Quality:  Transition from legacy silos to a  Unified Person-Centric Search , treating name, phone, email, and location as joint evidence rather than separate search lanes.
* Heuristic Iteration:  Utilize  Learning-to-Rank (LTR) Versioning  to trace and deploy improving entity resolution (ER) logic based on automated evaluation metrics.

4. LLM Orchestration: Progressive Tool Disclosure & Skill-Based Routing
The framework replaces legacy "single-pass" LLM calls with a  Two-Pass Routing  architecture to maximize token efficiency and prevent instruction drift.Execution Logic:
* Pass 1: Tool Router:  The system MUST send a lightweight routing prompt (~248 tokens) containing 1-line tool summaries. The LLM returns a JSON array of required "skills."
* Pass 2: Execution:  Only the full schemas and rich, step-by-step "skill" instructions for the selected tools are loaded into the context.
* Enrichment Bypass:  System MUST bypass the router for enrichment workflows. The LLM receives all tools upfront to plan multi-step execution paths.Token Comparison by Scenario:  | Scenario | Legacy Single-Pass (All Tools) | New Skill-Based (Routed) | Token Savings | | :--- | :--- | :--- | :--- | | Phone Search | ~21,000 chars | ~293 chars + routing |  ~99%  | | Name Search (Chat) | ~21,000 chars | ~3,306 chars + routing |  ~84%  | | Employment Search | ~21,000 chars | ~6,130 chars + routing |  ~71%  |

6. Data Integration & Persistence Layer
* Inference:  Gemini 2.5 Flash / 2.0 Flash (optimized for function calling).
* Primary Database:   PostgreSQL  with  pgvector  for semantic search over chat history.
* Caching:   Redis/Valkey  (Standalone for dev; Cluster for production).
* Identity Key:  spk_person_id.
* Critical Defect:  59.3% of unique mart rows currently fall back to name-string identifiers because entity resolution failed at the mart build layer.
* Constraint - Data Independence:  Architectural Warning: Vendors  22_01_01  (PDL Primary) and  22_01_06  (PDL Experience) are not independent; both are subsets of the same bulk delivery. LTR models MUST NOT treat corroboration between these two as an independent consensus signal.
7. Evaluation Framework & Behavioral Analytics
Since external ground truth is unavailable, the framework utilizes internal metrics to assess quality.Internal Metrics (v3):
1. Held-out Vendor Prediction:  Train on  $N-1$  vendors; test against the  $N^{th}$  vendor for overlapping persons.
2. Cross-vendor Consensus:  Average vendor count corroborating the top-1 ranked employer.
3. Temporal Plausibility:  Fraction of top picks where is_current=1.
4. Stability under Perturbation:  Rate of top-pick changes when a vendor is excluded.Interaction Sequence & Psycholinguistic Modeling:  The system MUST implement a  Markov Transition Model  to predict search failure.
* Analytic Thinking (Linguistic Feature):  High analytic thinking in a prompt indicates structured identifiers; low thinking/high emotional tone indicates a  "Stuck User" loop .
* Trajectory Analysis:  Cluster sequences (e.g., NAME_SEARCH -> REFORMULATE -> REFORMULATE). Transition probabilities MUST be used to detect repair loops and trigger automated clarifying questions (e.g., "Would you like to add a city or state?") to resolve ambiguity before abandonment.
8. Implementation Roadmap: The Independence Plan
The objective of the Independence Plan is to achieve a vendor-independent LTR pipeline, eliminating the "Golf" mart dependency.
* WC-01:   Primary Blocker:  Implementation of the person_links join within claims_loader.py to resolve raw vendor parquets to spk_person_id.
* WC-02:  Integration of temporal signals (start/end dates) into the feature engineering pipeline.
* WC-03:  Development of an independent source contract for raw vendor parquets.
* WC-04–WC-06:  Evaluation and gating of employer entity resolution bridges.
* WC-07:  Addition of "Golf chronology rank" as a feature for ensemble evaluation.
* WC-08:  Verification of the vendor-direct path against resolved identity keys. (Note: Current v2 prototype fails on multi-vendor entity pct; WC-08 MUST iterate on bridge logic).
