What follows is a design walkthrough for an autonomous heuristic discovery framework, built to transition entity resolution from siloed search buckets into an integrated, agent-driven loop. The architecture commits to token efficiency through skill-based routing, retrieval quality via a unified person-centric index, and continuous heuristic iteration using learning-to-rank versioning. There is no manual intervention at runtime; the point is to let the orchestration layer synthesize disparate data signals into a canonical entity view automatically.

The core orchestration relies on a two-pass routing layer rather than injecting the entire tool surface into the context window. The system passes a lightweight prompt of approximately 248 tokens, containing one-line tool summaries, to a routing agent. The agent returns an array of required skills, and the second pass loads only the full schemas and detailed instructions for those selected tools.

The token savings are substantial. The table below outlines the token reduction achieved by the two-pass routing architecture compared to the legacy single-pass method.

| Scenario | Legacy Single-Pass | New Skill-Based (Routed) | Token Savings |
| :--- | :--- | :--- | :--- |
| Phone Search | ~21,000 chars | ~293 chars + routing | 99% |
| Name Search | ~21,000 chars | ~3,306 chars + routing | 84% |
| Employment Search | ~21,000 chars | ~6,130 chars + routing | 71% |

I considered leaving the router in place for data enrichment tasks and decided not to: enrichment workflows require multi-step planning, and passing all tools upfront allows the agent to build the full execution path at once.

The history of the learning-to-rank versioning illustrates the difficulty of cross-vendor consensus. The initial raw-vendor path failed entirely, generating zero cross-vendor overlap because the identifiers were prefixed with vendor-specific strings. Records from different sources never shared a key, making consensus mathematically impossible. The architecture now requires all raw vendor data to be joined against a centralized person-link table to resolve records to a canonical person identifier before any ranking occurs.

Heuristics are authorized for production through a four-gate waterfall evaluation process. The framework tests each proposed data bridge against a strict threshold before it is allowed to influence the entity resolution graph. 

- **Gate 1 (Network Bridge):** Requires a fill rate above 50 percent for primary vendors.
- **Gate 2 (Persistent ID Bridge):** Requires a fill rate above 70 percent.
- **Gate 3 (Stock Ticker Bridge):** Sparse bridge retained for public company records.
- **Gate 4 (Fuzzy Match Bridge):** Requires a false merge rate below 5 percent.

The fuzzy match bridge was authorized only after its false merge rate was confirmed at 2.6 percent, keeping it under the 5 percent ceiling. It operates entirely within a low-confidence tier.

Because external ground truth is unavailable for these entities, the system relies on internal metrics for evaluation. The framework trains on the first $N-1$ vendors and tests prediction against the final vendor, measuring cross-vendor consensus and temporal plausibility.

To monitor the interaction sequences, the system implements a Markov transition model to predict search failure. The model classifies psycholinguistic features in the prompt: high analytic thinking correlates with structured identifiers, while low analytic structure paired with high emotional tone indicates a user caught in a repair loop. Transition probabilities detect these failure patterns, triggering automated clarifying questions to resolve the ambiguity before the sequence is abandoned. The architecture does not commit to a static set of rules; what it offers is a mechanism for continuous heuristic discovery that corrects itself as new vendors and signals enter the graph.
